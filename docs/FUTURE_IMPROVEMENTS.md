### FI-01：候选扩展方向

一期保持 `Dense + BM25 + LightGCN + Popularity -> weighted RRF -> 硬过滤`
的范围。后续可按优先级验证：

1. 增加 Cross-Encoder 或 ColBERT 重排，提高问题与候选商品的精细匹配能力。
2. 使用元数据中的 `bought_together` 构建共购图，补充配件和互补商品召回。
3. 增加查询改写与中英文同义词扩展，提高自然语言问题的召回率。
4. 对比 SPLADE 等 Neural Sparse 方法，验证是否优于基础 BM25。
5. 有查询相关性标注或真实搜索日志后，再训练 Learning to Rank 排序器。
6. 后续如需强化近期行为建模，再评估 SASRec 等序列推荐模型。

### FI-02：ESCI-US 搜索评估与 Dense 召回模型微调

- 状态：待执行
- 优先级：中
- 适用阶段：完成阶段 6 E5 与 BLaIR 商品检索基线后

#### 阶段 6 边界

一期阶段 6 使用预训练的 `intfloat/e5-small-v2` 和
`hyp1231/blair-roberta-base` 分别构建商品 Dense 索引，完成零微调对比。
阶段 6 不微调模型参数，也不接入 Amazon ESCI 数据集或 ESCI 评估代码。

阶段 6 仍按 `docs/DEVELOPMENT_STEPS.md` 验收，仅使用 BM25、E5、BLaIR 和少量人工
编写的英文查询检查常规召回效果。中文输入仅通过最小 LLM 翻译层转换为英文，再进入
相同检索链路。ESCI 相关工作全部作为阶段 6 完成后的独立扩展。

Amazon Reviews 2023 `Musical_Instruments` 数据包含用户交互、商品元数据和评论证据，
但不包含用户搜索词和查询-商品相关性标注，无法直接构造标准的 Dense 召回训练数据：

```text
(query, positive_product, negative_product)
```

#### ESCI-US 数据口径

后续扩展暂定使用 Amazon ESCI Shopping Queries Dataset。只使用
`product_locale == "us"` 的美国站数据，不使用 `es`、`jp` 或多站点合计口径。

ESCI 提供查询、商品文本和人工标注的相关性标签：

```text
Exact
Substitute
Complement
Irrelevant
```

使用以下链路将 ESCI 商品精确关联到 CartWise 目录：

```text
ESCI.product_id
-> Amazon Reviews 2023 review.asin
-> CartWise parent_asin
```

允许增加 `product_id == parent_asin` 的直接回退。不要根据商品标题自动转移相关性标签，
标题模糊匹配仅用于人工复核。

当前已生成可复现的重叠分析脚本：

```text
scripts/experiments/download_esci_examples.py
scripts/experiments/analyze_esci_overlap.py
```

当前 CartWise 目录包含 `24,587` 个 `parent_asin`。ESCI-US 精确关联覆盖情况：

| 范围 | query-product 人工标注数 | 查询数 | 匹配 `parent_asin` 数 | 目录覆盖率 |
|---|---:|---:|---:|---:|
| `us/train` | 5,837 | 773 | 3,185 | 12.95% |
| `us/test` | 1,732 | 243 | 1,274 | 5.18% |
| `us/train ∪ us/test` | 7,569 | 1,016 | 3,769 | 15.33% |

#### FI-02A：零微调搜索评估

在模型完全未使用 ESCI 数据训练或调参时，将关联后的：

```text
us/train ∪ us/test
```

合并为：

```text
CartWise ESCI-US overlap evaluation set
```

该集合用于对比预训练 `e5-small-v2`、`hyp1231/blair-roberta-base`、
BM25 和 Hybrid Retrieval 的零微调搜索排序效果。报告中必须明确：它不是 ESCI
官方测试集，而是 CartWise 乐器目录与 ESCI-US 的重叠评估集。

同时永久保留关联后的 `us/test` 交集。后续只要使用 ESCI 数据训练、选择模型或调参，
就不能继续将合并集作为最终测试集，必须只在未参与训练和调参的 `us/test` 上报告最终
指标。

ESCI 只标注每个查询对应的一组候选商品，并未标注 CartWise 全部目录。因此该集合适合
评价候选排序，不应将结果表述为全目录 Dense 召回的严格 `Recall@K`。

#### FI-02B：Dense 模型微调

后续将 Dense 召回模型微调作为独立实验。训练时优先将 `Exact` 作为强正样本，将
`Substitute` 作为弱正样本或次级正样本，并根据实验口径将 `Complement` 和
`Irrelevant` 用作负样本。优先从同一查询对应的候选中构造困难负样本，避免只使用
过于简单的随机负样本。

优先采用以下实验顺序：

```text
ESCI-US 全量 train
-> 微调通用英文商品搜索 Dense 模型
-> 使用关联后的 us/train 做可选乐器领域适配
-> 使用关联后的 us/test 做最终英文乐器领域评估
-> 使用少量中文输入检查 LLM 翻译后的英文检索链路
-> 与原始 e5-small-v2 和 blair-roberta-base 基线比较
```

关联后的 `us/train` 只有 `773` 个查询和 `5,837` 条人工标注，适合小规模领域适配和
可行性验证，不应作为从头训练 Embedding 模型或证明通用商品搜索效果的唯一数据源。

#### 乐器领域数据构造备选方案

Amazon ESCI 官方商品表不提供商品类目字段，也没有官方
`Musical_Instruments` 子集。乐器领域数据保留以下三种构造方案：

| 方案 | 方法 | 用途与限制 |
|---|---|---|
| 方案 A：文本关键词筛选 | 根据查询、标题和描述中的 `guitar`、`tuner`、`microphone`、`drum`、`keyboard`、`violin` 等词筛选 ESCI 数据 | 成本低，适合快速实验；存在歧义和漏召回，不能视为严格类目子集 |
| 方案 B：使用 ESCI-S 补充类目 | 使用第三方 ESCI-S 的 `category` 字段筛选 `Musical Instruments` 类目路径 | 类目筛选较准确；ESCI-S 不是 Amazon Science 官方数据，需要记录来源、版本和覆盖率 |
| 方案 C：与 Amazon Reviews 2023 关联 | 使用 ESCI `product_id` ASIN 关联 Amazon Reviews 2023 的 `asin`，映射到 `parent_asin` 后筛选 `Musical_Instruments` | 最适合 CartWise 乐器领域验证；关联覆盖不完整，必须统计匹配率和缺失率 |

如果后续实验表明领域差异明显，再评估使用乐器子集继续微调。不要在缺少覆盖率统计和
独立评估集时，直接将文本关键词筛选结果视为高质量领域训练集。

#### 验收指标

候选排序评估至少记录：

```text
NDCG@10
NDCG@20
MRR
Precision@10
Success@10
```

只有在后续建立全目录相关性评估协议后，才额外记录：

```text
Recall@10
Recall@20
```

同时记录：

```text
ESCI 各标签样本数
乐器领域子集查询数与商品数
方案 C 的 ASIN 关联覆盖率和缺失率
索引构建耗时
查询 P95 延迟
```

同时保留预训练 `e5-small-v2` 和 `blair-roberta-base` 的零微调指标，
避免只报告微调后的结果。

### FI-05：LightGCN 一对多负采样消融实验

- 状态：待执行
- 优先级：高
- 适用阶段：完成基础 LightGCN 训练与评估后

#### 当前方案

一期 LightGCN 基线对每条正样本边均匀随机采样 `1` 个用户未交互商品，
使用 `1:1` 的正负样本比例训练 BPR loss。每个 epoch 重新采样负样本。

当前阶段保留该实现，不引入 mini-batch 训练，也不引入 batch 内负采样，
以便先固定可复现的标准基线。

#### 实验方案

保持图结构、正反馈口径、随机种子、模型参数和评估流程一致，仅调整每条正样本边
对应的均匀随机负样本数量：

| 实验 | 正负样本比例 | 目的 |
|---|---:|---|
| A：当前基线 | `1:1` | 保留标准 BPR 基线 |
| B：一对三负采样 | `1:3` | 验证增加负样本是否提高排序质量 |
| C：一对五负采样 | `1:5` | 验证继续增加负样本是否仍有稳定收益 |

负样本必须排除用户已经交互过的商品。该消融实验先使用普通均匀随机采样，
不同时叠加困难负样本、热门度偏置采样、mini-batch 或 batch 内负采样，
避免无法判断指标变化来自哪项策略。

#### 评估指标

至少记录：

```text
Recall@10
NDCG@10
HitRate@10
每个 epoch 耗时
总训练耗时
GPU 峰值显存
```

#### 决策规则

- 优先验证 `1:3`。如果验证集排序指标稳定提升且训练成本可接受，再验证 `1:5`。
- 仅当 `1:5` 相比 `1:3` 仍有稳定收益时，才增加默认负样本数量到 `5`。
- 如果增加负样本没有稳定提升，则保留 `1:1` 基线，不提前引入更复杂的采样策略。

### FI-06：LightGCN 正反馈口径消融实验

- 状态：待执行
- 优先级：中
- 适用阶段：完成基础 LightGCN 训练与评估后

#### 背景

当前使用 Amazon Reviews 2023 `Musical_Instruments` 官方 `5-core`
Leave-Last-Out 划分构建用户-商品二部图。交互表示用户提交过评分或评论，
不是点击日志，也不能全部严格表述为购买记录。

现有数据质量报告显示，`5core` 的 `511,836` 条交互均可关联回原始评论，
其中 `461,477` 条交互的 `verified_purchase=true`，占 `90.16%`。
因此，直接使用官方划分适合作为一期基线，但低评分交互可能为偏好学习引入噪声。

#### 实验方案

保持模型结构、随机种子、训练参数和评估流程一致，对比：

| 实验 | LightGCN 正样本边 | 目的 |
|---|---|---|
| A：官方全量交互 | 使用官方 `5-core` 全部交互 | 保留标准基线，预测未来交互行为 |
| B：高评分交互 | 仅使用 `rating >= 4` 的交互 | 验证过滤低评分后是否更适合学习正向偏好 |

过滤后需要重新检查每位用户的有效历史长度，并按时间重新生成训练集、验证集和
测试集，避免将低评分验证或测试目标视为推荐成功。

#### 评估指标

至少记录：

```text
Recall@10
NDCG@10
Recall@20
NDCG@20
```

同时记录过滤后的用户数、商品数、训练边数、验证边数和测试边数，避免仅比较指标
而忽略数据规模变化。

#### 决策规则

- 如果实验 B 在验证集和测试集上稳定优于实验 A，则后续推荐模型默认使用
  `rating >= 4` 的正反馈口径。
- 如果实验 B 没有稳定提升，则保留官方全量交互方案，并在报告中说明其语义是
  “预测未来评分或评论交互”，而不是“预测未来购买”。

### FI-07：RAG 评论证据容量消融实验

- 状态：待执行
- 优先级：中
- 适用阶段：完成基础评论证据检索后

#### 当前方案

一期先使用每个商品最多 `10` 条评论作为 RAG 证据候选，其中最多优先保留
`2` 条 `rating <= 2` 的低评分评论。其余候选优先保留已验证购买、
`helpful_vote` 较高和时间较新的评论。

该方案记为：

```text
10-2：每个商品最多保留 10 条评论，其中最多优先保留 2 条低评分评论
```

#### 背景

当前 `5core` 商品目录包含 `24,587` 个商品，对应 `2,119,353` 条原始评论。
每个商品平均有 `86.20` 条评论，中位数为 `32` 条，P25 为 `17` 条。
当前最终保留 `240,525` 条评论证据候选，平均每个商品约 `9.78` 条。

这说明大部分商品会触发 `10` 条上限。该限制适合一期控制索引规模，但可能丢失
尺寸、安装、耐用性和适用场景等细分问题的长尾证据。

#### 实验方案

保持检索模型、分块策略和评估问题一致，对比：

| 实验 | 每个商品最多保留评论数 | 最多优先保留低评分评论数 |
|---|---:|---:|
| A：`10-2` | 10 | 2 |
| B：`20-4` | 20 | 4 |
| C：`30-6` | 30 | 6 |

低评分评论定义为 `rating <= 2`。增加评论数量时同步提高低评分评论上限，
保持优缺点证据的大致比例。

#### 评估指标

至少记录：

```text
证据覆盖率
引用一致性
索引条目数
索引构建耗时
查询 P95 延迟
```

人工验收问题应覆盖耐用性、尺寸、安装难度、适用人群和常见缺点等不同角度。

#### 决策规则

- 一期默认使用 `10-2`，不提前扩大索引。
- 如果 `20-4` 明显提高证据覆盖率，且索引规模与查询延迟仍可接受，则后续切换为
  `20-4`。
- 仅当 `30-6` 相比 `20-4` 仍有明确收益时，才进一步提高到 `30-6`。

### FI-08：BM25 持久化倒排索引优化

- 状态：待评估
- 优先级：中
- 适用阶段：完成阶段 6 BM25 基线并测量查询延迟后

#### 当前方案

阶段 6 使用 `rank-bm25` 建立本地 BM25 基线。离线构建脚本将商品 ID 和共享商品文档
保存为压缩 JSON：

```text
artifacts/indexes/product_bm25/<scope>/bm25.json.gz
```

审核工具或服务启动时读取该文件，根据本地商品文档重新分词，并初始化
`BM25Okapi` 的文档词频、文档长度、平均文档长度和 IDF 统计结构。同一进程内的后续
查询复用已经加载的内存结构，不会重复读取商品 Parquet 或重新拼接商品文档。

当前文件实际是可重建 BM25 内存结构的语料快照，不是持久化倒排索引。
`rank-bm25` 查询时会针对每个查询 token 扫描全部商品的词频字典，复杂度近似为：

```text
O(query token 数量 × 商品数量)
```

一期目录约有 `24,587` 个商品。当前方案实现简单、便于人工审核，先保留为阶段 6
基线，不提前增加自定义索引实现。

#### 后续任务

先在完整商品目录上测量预热后的 BM25 查询延迟。如果查询延迟、并发能力或服务启动
耗时不满足网页链路要求，再改为真正的持久化倒排索引：

```text
token -> [(document_id, term_frequency), ...]
token -> idf
document_id -> parent_asin
document_id -> document_length
average_document_length
BM25 parameters: k1, b
schema_version
```

查询时仅遍历查询 token 对应的 posting list，对命中文档累加 BM25 分数并取 Top K，
避免扫描完整商品目录。优先使用明确、可校验、可版本化的本地格式，不直接持久化
来源不透明的 `pickle` 对象。


### FI-09：推荐结果后的商品评论追问模式

- 状态：待执行
- 优先级：中
- 适用阶段：完成阶段 8 评论证据检索和阶段 9 LLM 解释生成后

#### 动机

一期主交互是“根据需求找商品”，RAG 主要用于为推荐结果提供评论证据和中文解释。
为了增强 RAG 交互深度，后续增加用户围绕已推荐商品进行评论级追问的能力，例如：

```text
第 2 个稳定吗？
有没有人说它不好安装？
这几个里面哪个更适合桌面空间小的情况？
这些商品的差评主要集中在哪？
```

该功能不改变商品召回、候选融合和硬过滤流程。商品仍由主推荐链路决定，评论追问只在
已经展示过的商品范围内检索证据并生成回答。

#### 实验方案

新增会话级追问接口或内部编排能力：

```text
session_id
selected_parent_asin 或 result_rank
question
```

处理流程：

```text
定位当前会话中已展示的商品
-> 将 result_rank 解析为 parent_asin
-> 仅检索该 parent_asin 对应的评论证据
-> 每个商品返回 Top K 条相关评论
-> LLM 基于这些评论回答追问
-> 校验输出中的 review_id 和 parent_asin
```

对于比较型追问，例如“这几个里面哪个更适合桌面空间小的情况”，只允许在当前会话最近
一批推荐商品中检索评论和比较，不允许引入新的商品候选。

LangChain 可作为可选编排层，仅用于 Prompt 模板、LLM 调用和输出解析。评论检索范围、
`review_id` 生成、`parent_asin` 归属校验和引用合法性校验必须保留在 CartWise 自有
代码中。

#### 验收指标

至少记录：

```text
追问回答引用合法率
引用评论 parent_asin 归属正确率
无关商品泄漏次数
LLM 输出 JSON 合法率
追问端到端 P50、P95 延迟
人工抽样的回答支持率
```

#### 决策规则

- 只有当引用合法率和商品归属正确率稳定达到 `100%` 时，才接入前端演示。
- 如果 LLM 输出无效、引用不存在或引用不属于对应商品，必须使用模板回退。
- 不允许追问模式绕过硬过滤器推荐新的商品；需要新商品时应触发正常推荐流程。

#### 决策规则

- 如果完整目录上的预热查询 P95 和网页端总延迟满足要求，保留当前简单实现。
- 如果 BM25 成为本地检索链路的明显瓶颈，或商品目录增长导致延迟明显上升，再实现
  持久化倒排索引。
- 升级时保持阶段 6 的商品文档构建顺序、分词规则和 BM25 参数不变，先验证数据结构
  优化本身，避免同时改变召回口径。
