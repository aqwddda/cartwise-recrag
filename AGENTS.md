# CartWise 开发规则

CartWise 是一个基于 RAG 的可解释个性化电商导购系统。所有开发应优先保证阶段边界清晰、实现可复现、数据口径一致，并避免提前引入当前阶段未要求的复杂功能。

## 基本开发原则

- 严格按照 `docs/CURRENT_STAGE.md` 和 `docs/DEVELOPMENT_STEPS.md` 逐阶段开发。
- 每次只完成当前阶段任务，不要提前实现后续阶段或未来改进功能。
- 每次开发前先阅读已有模块代码，优先复用现有接口、工具函数和数据口径，避免重复实现。
- 保持模块职责明确、依赖关系清晰，不要让 LangChain、LLM 或 RAG 逻辑接管推荐召回、硬过滤、候选融合、API 路由或会话状态。
- 不要覆盖、撤销或重写与当前任务无关的改动。

## 项目文档使用规则

开始任何开发、修改文件、运行测试或提交代码之前，必须先读取 `docs/CURRENT_STAGE.md`，确认当前阶段、当前任务、已完成内容、当前卡点和下一步操作。

完成较大的代码改动或阶段性任务后，必须更新 `docs/CURRENT_STAGE.md`，同步已完成内容、关键改动、验证结果、遗留问题和下一步操作。小范围修复或纯格式调整无需更新。

当前阶段的具体实现要求以 `docs/CURRENT_STAGE.md` 为第一依据，并参考 `docs/DEVELOPMENT_STEPS.md` 中对应阶段的文件清单、功能要求、验收标准和提交说明执行。不要跨阶段实现未要求的功能。

涉及项目整体架构、模块职责、数据范围或接口设计时，可以参考 `docs/PROJECT_PLAN.md`。涉及已经确定的技术选择、范围边界或实现口径时，以 `docs/DECISIONS.md` 为准。

`docs/FUTURE_IMPROVEMENTS.md` 只记录后续实验和二期改进。除非用户明确指定某个 FI 编号，否则不得主动实现其中的未来功能。

`docs/PROJECT_CONTEXT.md` 只作为项目背景和简历描述参考，不作为当前开发任务的主要依据。

如果文档之间存在冲突，应停止相关实现，向用户说明冲突点，并等待用户确认。

## 命令与环境规则

- 所有命令默认从仓库根目录运行。
- Python 命令统一使用 `.\.venv\Scripts\python.exe`。
- 当前用户位于中国大陆，并通过 VPN 访问网络。下载依赖、模型和数据时使用代理 `http://127.0.0.1:9508`。
- 访问 `127.0.0.1` 和 `localhost` 时必须绕过代理。
- 读取或写入中文文档时统一使用 UTF-8，避免 Windows 终端编码导致乱码或误判。

## Git 与文件安全规则

- 修改文件、执行可能影响外部状态的命令或提交代码前，必须先运行 `git status`。
- 禁止批量删除文件或目录。如果确实需要删除多个文件，应停止操作，向用户说明原因和待删除范围，由用户确认后手动删除。
- 不要提交 `.env`、数据文件、索引文件、模型权重、缓存文件或真实 API Key。
- 执行会重建索引、重建 Qdrant collection、覆盖模型产物或改变外部状态的操作前，必须先询问用户确认。

## CartWise 实现边界

- Dense 召回默认使用 E5。
- 新增召回通道时必须接入统一召回审核工具，并支持生成供人工审核的 HTML 报告和机器可读的 JSON 报告。
- 评论证据 RAG 与商品召回必须保持边界清晰。评论 RAG 只能基于 fusion 传入的候选商品检索证据，不能引入新的商品候选。
- LangChain 只用于评论证据 RAG，不接管商品召回、硬过滤、候选融合、API 路由或会话状态。
- 如果发现适合长期复用的新规则，应先询问用户是否写入 `AGENTS.md`，不要自行追加。

## 回复规则

- 如果本轮对话调用了任何 Skill，必须在回复末尾列明本轮使用的 Skill 名称，便于用户追踪和确认。
